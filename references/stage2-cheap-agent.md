## 输入边界

只读取 `phase1_handoff.json` 和本文件。包含详情页任务时再读取 `detail-page-framework.md`。默认不得遍历产品目录、打开原图、重新总结竞品或重做产品策略。

先运行 `validate_handoff.mjs`。结构错误、资产引用缺失或关键字段为空时，列出具体缺口并停止；不要用猜测补齐。

## 允许与禁止

允许：

- 根据每个任务的 `objective`、`copy_brief` 和 claim 生成消费者可见文案；
- 根据 `visual_brief`、资产摘要和 reference binding 生成当前页的画面描述；
- 对普通措辞进行精简和零风险改写。

禁止：

- 修改产品事实、资质、规格、链接方向、页面角色或参考图选择；
- 新增交接包不存在的数字、功效、材质或背书；
- 把竞品素材设为 API 上传参考；
- 把内部动作语言写进可见文案；
- 把设计说明、构图描述、卖点提取过程、参考图用途、注意事项、复核提醒或合规限制写进主图可见卖点；
- 重新读取全部图片来“验证”阶段一。

## 每个任务的输出

只输出一个新的最小 JSON，不得复制 `product`、`claims`、`assets`、`links`、原任务骨架或其他阶段一内容。每个任务只包含：

- `task_id`：必须与交接包一致；
- `visible_copy.headline`：大主标题；
- `visible_copy.subtitle`：强点击理由或消费者结果；
- `visible_copy.cards`：默认 3–4 条辅助卖点；
- `visible_copy.footer`：转化、规格或信任条；
- `prompt_delta.composition`：只写当前页构图差异；
- `prompt_delta.visual_evidence`：只写当前页必须表现的视觉证据，字符串数组；
- `prompt_delta.page_specific_constraints`：只写当前页特有约束，字符串数组；
- `review_notes`：仅记录真实缺口或风险，必须是字符串数组。

禁止在阶段二输出中出现 Windows 绝对路径、参考图文件名、完整产品锁定、全局视觉母版或通用禁用词。脚本会从阶段一交接包合并这些内容。

## 文案规则

- 假设文案会直接印在淘宝/天猫图片上，必须给消费者事实、场景代入或购买理由。
- 不得出现“讲清、直接说明、不含糊、围绕展开、证明价值、使用理由成立”等策划语言。
- 成分页写“成分事实 + 当前场景价值”；材质页写“材质事实 + 可感知结果”；尺寸页写“真实尺寸 + 对使用的帮助”。
- 不使用绝对化、治疗化、恐吓式表达；所有数字必须来自绑定 claim。
- 配方、成分、菌株、专利和检测类内容必须来自清晰可核验事实；粗抽文本、OCR、旧链接文案、文件名或待人工复核字段不得写入可见文案，只能写入 `review_notes` 或当前页限制。
- 配方、资质、检测报告、执行标准和标签信息不能写成“编号清晰、报告明确、执行标准明确、标签清晰、检测结果摆出来”等内部核对型卖点。报告类素材通常只作为样张或关键指标局部的视觉佐证；可见文案要回到消费者购买理由，例如柔软、好用、好收纳、常备安心、场景更顺手，而不是陈述所有竞品都应具备的基础要求。
- 主图可见文案不得写“植物乳杆菌配方线索”“配方线索”“配方清新双在线”“使用前看说明”“不作/不作为治疗承诺”等内容；这些属于未核实信息、含糊表达或合规控制，不是主图卖点。
- 合规边界只写在 `prompt_delta.page_specific_constraints` 或 `review_notes`，例如“不得生成治疗承诺、不得出现具体菌株名称”，不要把“不是治疗承诺”做成画面文字。
- 主图交付前必须专门审核 `visible_copy`，以及编译后提示词中“必须逐字显示”的四段文案。凡是更像给设计师、给模型、给审核人的话，都要移出可见文案，改放到构图、视觉证据、当前页限制或复核备注。
- 五张主图都包含主标题、辅助解释和底部条，但不要堆字到影响产品与主标题识别。
- 多颜色产品的 SKU 任务必须是一色一图。若阶段一交接包把所有颜色放在同一个 SKU 任务里，阶段二应返回缺口或先修正交接包；不要把全色系集合写成单张 SKU 图提示词。

## 链接视觉差异

- 读取每条链接的 `visual_direction`，并在同链接的主图、SKU 图和详情页中保持一致的场景氛围、主色/辅助色、光感和信息组件。
- 不同链接的提示词必须体现不同宣传方向：例如便携出行可偏轻透水感和包袋/通勤场景，口味选择可偏水果色和清爽陈列，成分/护理方向可偏克制科技感。不得把所有链接都写成同一套蓝白背景。
- 如果交接包里多条链接的 `visual_direction` 缺失、重复或无法区分，停止阶段二并返回缺口，不自行用全局视觉母版补成同一种风格。

## 提示词增量规则

`prompt_delta`只写差异，不生成完整提示词。最终发给 API 的提示词由编译脚本合并，且必须保持极简结构：

1. `composition`：主体位置、场景和信息卡组织，并落实当前链接的 `visual_direction`；
2. `visual_evidence`：材质微距、动作、结构拆解、参数卡等；
3. `page_specific_constraints`：只保留当前页风险，不复制全局规则。

不要写参考图路径，也不要重述参考资产用途。第一张身份基准、允许上传路径及竞品过滤由合并脚本处理。证照、备案或资料页如果使用上传参考图，`prompt_delta.composition` 要明确“用画框/展示框承载参考图中的证照/资料，文字编号印章和版式保持参考图原样”，但消费者可见文案仍要转化为购买理由；非医疗器械产品不要固定规划注册证或备案证明页。

最终 `生成提示词` 只允许包含：

```text
参考图：{说明已上传参考图如何使用}
画面规格：{1440x1440 或 1440x1920}
画面描述：{视觉构图、产品主体、标题区、卖点卡、背景与证据元素}
页面中需要带出的文字：
主标题：{...}
副标题：{...}
辅助信息：{...}
底部条：{...}
```

不要把任务编号、页面角色、前后承接关系、阶段一目标、参考图路径、执行模式、审核提醒、复核备注、通用禁用词或“不要写某某”放进 `生成提示词`。这些只留在结构化字段、复核备注或脚本逻辑里。
